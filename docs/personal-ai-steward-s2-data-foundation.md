# 私人智能管家 S2 个人数据底座

本文档承接 [S0 边界与安全基线](personal-ai-steward-s0-boundary-baseline.md) 和 [多阶段目标文档](personal-ai-steward-multi-stage-goals.md)，定义私人智能管家在 S2 阶段需要建立的个人数据底座。

S2 的重点不是让系统更“聪明”，而是先让事件、时间线、任务、意图、记忆、知识和审计日志进入统一、可追溯、可纠正、可导出、可同步演进的数据结构。后续 S3 三端同步、S4 自主执行、S5 子 Agent 调度都必须依赖这个底座。

## S2 目标与边界

### 阶段目标

S2 要把 S1 中的临时事件、任务和审计记录升级为稳定数据模型。完成后，用户应能在工作台中搜索、筛选、追溯、纠正和导出自己的个人数据，而不是只看到一堆不可解释的后台日志。

S2 必须解决：

- 事件如何进入时间线。
- 事件如何转成任务、意图、记忆或知识。
- 记忆如何保留来源、置信度、适用范围和纠错历史。
- 知识如何保留原始来源和导入记录。
- 审计日志如何覆盖数据修改、工具调用和权限变化。
- 删除、纠正和导出如何统一表达。
- 哪些字段需要为 S3 跨端同步提前预留。

### S2 允许实现的能力

- 核心实体的数据模型。
- 本地数据库迁移策略。
- 统一搜索和筛选。
- 时间线基础聚类。
- 意图候选池。
- 记忆新增、纠正、归档和删除。
- 知识条目导入和来源追踪。
- 事件转任务、转意图、转记忆、转知识。
- 数据导出和删除。
- 敏感字段标记。
- 审计日志与来源引用展示。

### S2 明确不做的能力

- 不做三端真实同步。
- 不做复杂自主决策。
- 不做跨设备上下文恢复。
- 不做自动生成工具。
- 不做子 Agent 调度。
- 不让模型自动覆盖用户明确输入。
- 不让未经确认的推断直接写入长期记忆。
- 不做高风险数据的自动采集和自动分析结论。

## 数据设计原则

- 来源优先：任何任务、意图、记忆、知识和结论都必须能追溯来源。
- 用户可纠正：用户纠正后的数据优先级高于模型推断和自动摘要。
- 软删除优先：删除行为先记录为可同步的删除状态，后续再做物理清理。
- 摘要与原文分离：高敏内容不应为了搜索或审计而被复制到多个表。
- 当前事实与历史版本分离：记忆和知识可以更新，但历史变更要可审计。
- 局部上下文优先：项目事实、设备事实、阶段性偏好不默认升级为全局记忆。
- 同步预留：所有核心实体都必须具备稳定 ID、版本、更新时间和删除标记。
- 安全分层：每条数据必须能挂接 S0 的权限级别和数据级别。

## 核心实体总览

| 实体 | 定位 | S2 主要用途 |
|---|---|---|
| Event | 原始事件或摘要事件 | 构成时间线和后续转化来源 |
| TimelineSegment | 聚合后的时间块 | 把分散事件组织成可理解的工作段或生活段 |
| Task | 已确认任务 | 承载手动任务和确认后的系统建议任务 |
| Intent | 候选意图 | 保存尚未确认的潜在任务、承诺或兴趣 |
| Memory | 长期事实 | 保存偏好、项目事实、决策、习惯和上下文 |
| KnowledgeItem | 知识条目 | 保存文档、网页、笔记、文件、会议摘要等资料 |
| SourceRef | 来源引用 | 连接任何结论与原始事件、文件、网页或工具输出 |
| AuditLog | 审计记录 | 记录工具调用、数据变更、权限变化和阻止行为 |
| DataTag | 标签与敏感标记 | 支持搜索、分类、敏感控制和生命周期管理 |

## 统一基础字段

所有核心实体至少应具备以下字段语义。具体数据库命名可在后续技术方案中确定，但字段语义不能缺失。

| 字段 | 说明 |
|---|---|
| id | 稳定 ID，后续三端同步不能因设备不同而改变 |
| type | 实体类型或子类型 |
| title | 可读标题 |
| summary | 简短摘要，不保存敏感原文 |
| status | 当前状态，例如 active、archived、deleted |
| source | 创建来源，例如 manual、collector、agent、import、tool |
| data_level | 对应 S0 的数据分级 |
| permission_level | 读写或生成该实体所需的权限级别 |
| created_at | 创建时间 |
| updated_at | 最近更新时间 |
| deleted_at | 软删除时间 |
| version | 本地版本号，后续同步冲突处理使用 |
| device_id | 创建或最近修改设备 |
| user_confirmed | 是否经过用户确认 |
| audit_id | 创建或最近修改对应的审计记录 |

## Event 事件

### 定位

Event 是个人数据底座的最小事实单元。它可以来自用户手动输入、低风险采集器、工具输出、导入文件或 Agent 建议，但在 S2 中不应直接等同于记忆或知识。

### 事件类型

- manual_note：用户手动记录。
- clipboard_summary：剪贴板摘要。
- file_change：授权目录中的文件变化。
- browser_link：手动导入或授权导入的网页链接。
- system_status：系统状态摘要。
- task_activity：任务创建、完成、取消、延期。
- tool_result：工具调用结果摘要。
- import_record：知识或文件导入记录。
- audit_marker：重要审计事件的时间线标记。

### 状态

- active：可见和可转化。
- hidden：用户隐藏，不进入默认时间线。
- archived：归档保留。
- deleted：软删除，不参与搜索和推荐。

### 转化规则

一个 Event 可以转化为：

- Task：用户确认某事件需要行动。
- Intent：系统判断可能有行动价值，但未确认。
- Memory：用户确认某事件代表长期事实。
- KnowledgeItem：事件关联资料值得保存为知识。
- TimelineSegment：事件被聚合到某个时间块。

转化必须生成 SourceRef 和 AuditLog。

## TimelineSegment 时间线片段

### 定位

TimelineSegment 用于把多个 Event 聚合成可理解的时间块，例如“上午处理某项目报错”“整理旅行资料”“晚上阅读某主题文章”。

### 聚合依据

- 时间接近。
- 同一项目、目录、链接、任务或应用。
- 用户手动合并。
- 标题或标签相似。
- 同一工具调用链。

### 片段字段

- 开始时间。
- 结束时间。
- 关联事件列表。
- 主标签。
- 关联任务。
- 关联知识。
- 关联项目或生活领域。
- 自动聚合置信度。
- 用户确认状态。

### S2 限制

S2 只做基础聚类和手动调整，不做复杂行为分析，也不把时间线片段自动升级为长期记忆。

## Task 任务

### 定位

Task 是用户已确认需要管理的事项。S2 中 Task 可以来自用户手动创建，也可以来自 Intent 转化，但不能由系统跳过确认直接创建高风险执行任务。

### 任务类型

- manual：用户手动创建。
- from_intent：候选意图确认后创建。
- from_event：事件转化。
- checklist：清单型任务。
- review：复盘或确认任务。
- data_cleanup：数据清理任务。

### 任务状态

- open：待处理。
- in_progress：处理中。
- waiting：等待外部条件。
- done：完成。
- canceled：取消。
- archived：归档。

### 任务字段

- 目标。
- 背景摘要。
- 来源引用。
- 优先级。
- 截止时间。
- 验收标准。
- 关联事件。
- 关联知识。
- 关联记忆。
- 风险级别。
- 权限需求。
- 审计记录。

### S2 限制

S2 只定义任务数据和手动状态流转，不实现真实自动执行和子 Agent 执行。

## Intent 候选意图

### 定位

Intent 是“系统认为用户可能想做，但尚未确认”的候选项。它是自主能力进入真实任务前的缓冲区。

### 意图类型

- commitment：承诺事项。
- follow_up：需要跟进。
- reminder_candidate：可能需要提醒。
- reading_candidate：稍后阅读。
- cleanup_candidate：可能需要整理。
- project_blocker：潜在项目阻塞。
- recurring_pattern：重复行为模式。

### 意图字段

- 候选标题。
- 推断原因。
- 置信度。
- 触发来源。
- 相关事件。
- 建议动作。
- 风险级别。
- 用户处理状态。

### 意图状态

- candidate：候选。
- accepted：已接受并转任务或知识。
- dismissed：已忽略。
- muted：同类意图静音。
- expired：过期。

### S2 限制

Intent 不能直接执行工具，不能直接写入长期记忆，不能直接对外发送内容。

## Memory 记忆

### 定位

Memory 是关于用户、项目、偏好、习惯、决策和长期上下文的事实或准事实。S2 的关键是防止“未经确认的推断污染长期记忆”。

### 记忆类型

- preference：用户偏好。
- project_fact：项目事实。
- decision：历史决策。
- workflow：工作流习惯。
- constraint：约束和禁忌。
- person_context：人物和关系上下文。
- device_context：设备和环境上下文。

### 记忆字段

- 记忆内容。
- 适用范围：全局、项目、设备、人物、时间段。
- 来源引用。
- 置信度。
- 用户确认状态。
- 最近验证时间。
- 失效条件。
- 替代或冲突记忆。
- 历史版本。

### 记忆状态

- draft：草稿，未进入默认检索。
- active：当前可用。
- disputed：存在冲突，等待确认。
- stale：可能过期。
- archived：归档。
- deleted：删除。

### 纠正规则

- 用户手动纠正优先于自动推断。
- 新记忆与旧记忆冲突时，旧记忆不得被静默覆盖。
- 冲突记忆必须进入待确认队列。
- 记忆删除和降权必须写入 AuditLog。

## KnowledgeItem 知识条目

### 定位

KnowledgeItem 保存资料本身或资料摘要，例如文档、网页、笔记、文件索引、会议摘要、代码片段。知识不是记忆，未经确认的知识不得直接变成“关于用户的事实”。

### 知识类型

- document：文档。
- webpage：网页。
- note：笔记。
- code_snippet：代码片段。
- meeting_summary：会议摘要。
- file_index：文件索引。
- imported_text：导入文本。
- report：Agent 或工具生成的报告。

### 知识字段

- 标题。
- 摘要。
- 原始来源。
- 本地文件位置或外部 URL。
- 导入方式。
- 内容 hash。
- 关联标签。
- 关联任务。
- 关联项目。
- 敏感级别。
- 是否允许进入检索索引。

### S2 限制

S2 可以保存知识条目和摘要，但不要求实现完整向量数据库、全文索引引擎或自动网页抓取。

## SourceRef 来源引用

### 定位

SourceRef 是 S2 的关键实体。它回答“这条任务、记忆、知识、意图或结论从哪里来”。

### 来源类型

- event。
- task。
- intent。
- memory。
- knowledge_item。
- audit_log。
- file。
- url。
- tool_result。
- manual_input。

### 来源字段

- 来源实体类型。
- 来源实体 ID。
- 原始位置，例如文件路径、URL、时间点、文本范围。
- 引用摘要。
- 可信度。
- 是否含敏感内容。
- 是否可展示给用户。

### 使用规则

- 任务、意图、记忆、知识都可以有多个 SourceRef。
- SourceRef 不复制敏感原文，只保存定位和摘要。
- 删除源实体后，引用必须显示来源已删除或不可用。

## AuditLog 审计日志

### 定位

AuditLog 记录系统对数据和工具的所有重要操作。S2 中审计不仅是安全要求，也是后续纠错、同步和复盘的基础。

### 审计范围

- 实体创建。
- 实体更新。
- 实体删除。
- 状态变更。
- 来源引用创建。
- 记忆纠正。
- 知识导入。
- 数据导出。
- 权限变更。
- 高风险操作阻止。

### 最小字段

沿用 S0 审计基线，并在 S2 中补充：

- 目标实体类型。
- 目标实体 ID。
- 变更前摘要。
- 变更后摘要。
- 变更原因。
- 用户确认状态。
- 是否可同步。

## DataTag 标签与敏感标记

### 定位

DataTag 为事件、任务、意图、记忆和知识提供统一分类能力。标签既用于搜索，也用于权限和生命周期管理。

### 标签类型

- 普通标签：项目、生活、学习、健康、财务、家庭、开发。
- 系统标签：自动导入、用户确认、待确认、待清理、已归档。
- 敏感标签：凭据、身份、医疗、财务、法律、位置、通讯。
- 生命周期标签：临时、长期、过期、待复核。

### 规则

- 敏感标签应影响默认展示、导出和检索。
- 用户可以手动添加、删除或覆盖自动标签。
- 自动标签必须保留来源和置信度。

## 实体关系

S2 推荐关系如下：

```text
Event -> TimelineSegment
Event -> Intent
Event -> Task
Event -> Memory
Event -> KnowledgeItem

Intent -> Task
Task -> SourceRef
Memory -> SourceRef
KnowledgeItem -> SourceRef

AuditLog -> Event
AuditLog -> Task
AuditLog -> Intent
AuditLog -> Memory
AuditLog -> KnowledgeItem
AuditLog -> PermissionChange
```

关键约束：

- Event 是事实来源，不等同于结论。
- Intent 是候选，不等同于任务。
- KnowledgeItem 是资料，不等同于用户事实。
- Memory 是长期上下文，必须有来源或用户确认。
- AuditLog 不应被普通实体关系绕过。

## 数据生命周期

### 创建

数据可以由用户手动创建、授权采集器创建、工具结果创建或导入创建。自动创建的数据默认应标记来源和置信度。

### 使用

数据被搜索、展示、转化、引用或用于生成建议时，必须尊重数据级别、权限级别和用户确认状态。

### 纠正

用户纠正应创建新版本或修正记录，不应静默覆盖旧数据。旧版本可归档，但必须可追溯。

### 删除

S2 使用软删除：

- deleted_at 记录删除时间。
- status 设置为 deleted。
- 默认搜索和推荐不再返回。
- 审计日志保留删除动作。
- 后续 S3 同步可以传播删除状态。

### 导出

导出必须支持：

- 按实体类型导出。
- 按时间范围导出。
- 按标签导出。
- 排除敏感内容。
- 包含来源引用。
- 包含审计摘要。

## 搜索与筛选

S2 统一搜索至少应支持：

- 标题和摘要搜索。
- 实体类型过滤。
- 时间范围过滤。
- 状态过滤。
- 标签过滤。
- 数据级别过滤。
- 来源类型过滤。
- 是否用户确认过滤。

S2 不要求完整语义搜索，但必须为后续语义检索保留索引入口。

## 与权限和审计的关系

S2 数据底座必须继承 S0 的权限与数据分级：

- 每个实体都有 data_level。
- 每个写入动作都有 permission_level。
- 每个敏感实体默认不进入外部模型上下文。
- 每个自动转化动作都创建 AuditLog。
- 每个记忆写入都需要 SourceRef 或用户确认。
- 每个导出动作都写入 AuditLog。

## 与 S3 同步的关系

S2 不实现同步，但必须为 S3 预留：

- 稳定 ID。
- version 字段。
- updated_at 字段。
- deleted_at 字段。
- device_id 字段。
- user_confirmed 字段。
- source 和 audit_id 字段。

进入 S3 前必须能表达：

- 新增。
- 更新。
- 删除。
- 纠正。
- 归档。
- 冲突。
- 用户确认。

## UI 入口要求

S2 对应工作台至少需要以下页面或视图：

- 时间线：查看 Event 和 TimelineSegment。
- 任务：查看 Task，并支持从 Event/Intent 转任务。
- 意图候选池：查看、接受、忽略、静音 Intent。
- 记忆库：查看、纠正、归档、删除 Memory。
- 知识库：导入、查看、标记、删除 KnowledgeItem。
- 来源面板：展示 SourceRef。
- 审计日志：展示关键数据变更和工具调用。
- 数据管理：导出、删除、敏感标记、标签管理。

## S2 完成验收

S2 完成时必须满足：

- 用户能查看事件、任务、意图、记忆、知识和审计日志。
- 用户能把事件转成任务、意图、记忆或知识。
- 用户能把意图接受为任务或忽略。
- 用户能查看一条记忆的来源。
- 用户能纠正一条记忆，旧版本不再作为当前事实使用。
- 用户能删除任意事件、任务、意图、记忆或知识。
- 用户能导出个人数据。
- 敏感数据有明确标记，不默认进入模型上下文。
- 所有核心实体都有稳定 ID、版本、更新时间和删除状态。

## 后续进入 S3 的门槛

进入 S3 三端常驻与同步前，必须确认：

- 核心实体的 ID、version、updated_at、deleted_at、device_id 已稳定。
- 删除和纠正行为能被同步模型表达。
- 冲突状态可以被 UI 展示，而不是静默覆盖。
- 敏感数据同步策略已经有明确默认值。
- 审计日志能记录跨设备相关字段。
- 本地导出和删除已经可用。
