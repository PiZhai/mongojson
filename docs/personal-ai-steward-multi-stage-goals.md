# 私人智能管家多阶段目标文档

本文档承接 [私人智能管家需求与目标方向](personal-ai-steward-requirements.md)，用于把长期愿景拆成可逐步交付、可验收、可暂停、可调整的阶段目标。

它不是详细技术方案，也不是任务排期表。它回答的是：每个阶段要解决什么核心问题、做到什么程度才算完成、哪些能力应该延后、进入下一阶段前必须守住哪些质量门槛。

## 总体目标

构建一个三端互通、长期常驻、高权限但可控、具备记忆与知识沉淀、能调用子智能体、能主动发现并执行任务的私人智能管家。

长期形态是“个人操作系统级智能层”：

- 它理解用户的设备、文件、项目、生活事项、知识材料和长期偏好。
- 它能把每天发生的事情重建成时间线。
- 它能把模糊意图变成候选任务。
- 它能在授权范围内执行真实操作。
- 它能通过记忆反省机制避免长期记错。
- 它能让多个子 Agent 协作完成复杂问题。
- 它能在重复工作中生成新工具，持续扩展自己的能力边界。

## 阶段划分总览

| 阶段 | 名称 | 核心问题 | 结果形态 |
|---|---|---|---|
| S0 | 产品边界与安全基线 | 先明确什么能做、什么不能自动做 | 可执行的边界、权限、数据和审计原则 |
| S1 | 单机本地管家原型 | 证明本地常驻、事件记录、任务入口可行 | Windows 单机可用的本地 Agent + Web 工作台 |
| S2 | 个人数据底座 | 让事件、任务、意图、记忆、知识有统一模型 | 可查询、可追溯、可纠正的数据系统 |
| S3 | 三端常驻与同步 | 让 Windows/macOS/Linux 互通上下文 | 三端设备身份、同步、冲突处理和状态面板 |
| S4 | 受控自主执行 | 从手动任务进入低风险自主任务 | 候选意图、自主任务、权限审批、审计闭环 |
| S5 | 子 Agent 与工具生态 | 让专业 Agent 和工具完成复杂任务 | 子 Agent 调度、工具注册、任务证据包 |
| S6 | 创造性智能层 | 实现传送门、自动工具、多代理会议等高级能力 | 可长期陪伴、可自我扩展的个人智能系统 |
| S7 | 稳定化与长期运营 | 让系统可信、可维护、可迁移 | 备份恢复、性能治理、数据治理、版本升级 |

## 跨阶段原则

- 每阶段都必须保留人工可控入口，不能只依赖自主行为。
- 每阶段都必须能解释数据来源、权限来源和操作结果。
- 每阶段都必须允许关闭自主能力，只保留记录、搜索和手动任务。
- 每阶段的新增能力都要能回退，不把实验能力混入不可逆核心。
- 高权限能力必须晚于权限中心、审计日志和模拟执行能力。
- 记忆必须晚于来源追踪和纠错机制，不能先堆不可解释的长期记忆。
- 三端同步必须晚于本地数据模型稳定，否则跨端冲突会放大错误。

## S0 产品边界与安全基线

### 阶段目标

把私人智能管家的能力边界、安全原则、数据边界和不可做事项先定下来，避免后续实现时把“高权限”做成不可控的全盘授权。

### 要解决的问题

- 哪些数据可以采集，哪些默认不采集。
- 哪些操作可以自动执行，哪些只能生成草稿。
- 哪些能力属于第一版，哪些属于远期愿景。
- 高权限操作如何分级、审批、审计和回滚。
- 当前工具项目中应该承担哪些入口。

### 关键交付物

- 总需求方向文档。
- 多阶段目标文档。
- [S0 边界与安全基线](personal-ai-steward-s0-boundary-baseline.md)。
- 权限分级草案。
- 数据采集分级草案。
- 高风险操作清单。
- 第一版非目标清单。

### 验收标准

- 能明确回答“第一版做什么、不做什么”。
- 能明确回答“哪些操作绝不自动执行”。
- 能明确回答“用户如何查看它做过什么”。
- 能明确回答“用户如何停止自主执行”。

### 暂不做

- 不做真实后台常驻。
- 不做跨端同步。
- 不接入真实系统权限。
- 不实现模型调度和子 Agent。

## S1 单机本地管家原型

### 阶段目标

在一台 Windows 设备上做出最小可用闭环：本地常驻进程采集低风险事件，Web 工作台展示事件、任务和基础对话入口。

### 用户价值

用户能第一次看到“我的设备上发生了什么、我记录了哪些任务、管家准备做什么”，并能通过当前工具项目与管家交互。

### 范围

- Windows 本地守护进程。
- 本地数据库。
- 当前项目中的管家工作台入口。
- 低风险事件采集。
- 手动任务创建。
- 基础执行日志。
- 最小权限配置。

### 低风险采集源

- 剪贴板文本摘要，默认不保存敏感字段原文。
- 指定目录的文件新增和修改事件。
- 浏览器链接手动导入。
- 手动添加的笔记、任务和知识条目。
- 系统基础状态：磁盘、网络、进程摘要、服务健康。

### 关键交付物

- [S1 单机本地管家原型](personal-ai-steward-s1-local-prototype.md)。
- 本地 Agent 启动和停止方式。
- Agent 心跳接口。
- 事件列表页面。
- 任务列表页面。
- 基础对话页面。
- 本地审计日志页面。
- 数据清理按钮。

### 验收标准

- Agent 重启后数据不丢。
- 用户能看到最近事件、最近任务和最近工具调用。
- 用户能手动创建、完成、取消任务。
- 用户能删除采集到的任意事件。
- 所有采集动作都能在审计日志中看到来源。

### 阶段门槛

进入 S2 前，必须确认：

- 本地数据库 schema 支持后续扩展。
- 事件、任务、审计日志的 ID 和时间字段稳定。
- 不存在默认采集敏感目录或私密浏览内容的行为。
- 工作台可以作为后续模块的主入口。

### 暂不做

- 不做三端同步。
- 不做自动执行命令。
- 不做邮件、聊天、日历深度接入。
- 不做自动删除、自动发送、自动付款等高风险操作。

## S2 个人数据底座

### 阶段目标

把事件、时间线、任务、意图、记忆、知识和审计日志统一成稳定数据底座。这个阶段重点不是更聪明，而是让后续所有智能能力都有可靠的数据承载。

### 用户价值

用户能搜索、筛选、追溯和纠正自己的个人数据，而不是只看到一堆临时日志。

### 核心实体目标

- Event：原始事件和摘要事件。
- TimelineSegment：聚合后的时间块。
- Task：手动任务和系统建议任务。
- Intent：尚未确认的候选意图。
- Memory：长期事实、偏好、决策和项目上下文。
- KnowledgeItem：文档、网页、笔记、文件、会议摘要。
- AuditLog：工具调用、数据修改、权限变化。
- SourceRef：任意结论对应的来源引用。

### 关键能力

- 统一搜索。
- 时间线基础聚类。
- 意图候选池。
- 记忆新增、纠正、删除。
- 知识条目导入。
- 来源追踪。
- 敏感字段标记。
- 数据导出。

### 关键交付物

- [S2 个人数据底座](personal-ai-steward-s2-data-foundation.md)。
- 数据模型文档。
- 数据库迁移策略。
- 时间线页面。
- 意图候选池页面。
- 记忆库页面。
- 知识库页面。
- 来源引用组件。
- 数据导出和删除能力。

### 验收标准

- 用户能问“这条记忆从哪里来”，系统能展示来源。
- 用户能把某个事件转成任务、知识或记忆。
- 用户能把某个候选意图转成正式任务或忽略。
- 用户能纠正一条记忆，后续检索结果不再使用旧版本作为当前事实。
- 用户能导出个人数据。

### 阶段门槛

进入 S3 前，必须确认：

- 所有核心实体都有稳定 ID。
- 所有可同步数据都有版本字段或变更时间。
- 删除和纠正行为能被同步模型表达。
- 敏感数据已经有本地加密或至少有清晰隔离方案。

### 暂不做

- 不做复杂自主决策。
- 不做跨设备上下文恢复。
- 不做自动生成工具。
- 不让记忆自动覆盖用户明确输入。

## S3 三端常驻与同步

### 阶段目标

让 Windows、macOS、Linux 三端都能运行本地 Agent，并通过私有同步机制共享必要数据、任务、记忆和知识。

### 用户价值

用户可以在任意设备上继续查看任务、时间线、知识和记忆，不需要重复解释上下文。

### 范围

- Windows/macOS/Linux 本地守护进程。
- 本地管理面与跨设备 Peer 协议面使用独立监听和最小路由集合。
- 设备注册和设备身份。
- 私有同步节点或点对点同步。
- 本地优先写入。
- 冲突检测。
- 同步状态面板。
- 设备级权限控制。

### 同步对象

- 任务。
- 意图。
- 记忆。
- 知识元数据。
- 时间线摘要。
- 审计摘要。
- 设备状态。
- 工具能力声明。

高敏内容可分阶段同步。第一版同步可以只同步摘要和索引，不同步原始敏感内容。

### 关键交付物

- [S3/S4 运行与验证基线](personal-ai-steward-s3-s4-runtime.md)。
- 三端 Agent 安装和启动说明。
- 设备身份模型。
- 同步协议草案。
- 同步状态页面。
- 冲突处理页面。
- 设备权限页面。
- 远程设备禁用或撤销能力。

### 验收标准

- 三端能分别启动并上报心跳。
- 在一台设备创建的任务能出现在另一台设备。
- 设备离线后重新上线能补齐变更。
- 冲突不会静默覆盖，必须进入冲突队列。
- 用户能撤销某台设备的同步权限。
- 局域网只能访问签名保护的 Peer 协议面，不能访问任务、记忆、自主设置或设备撤销等管理 API。

### 阶段门槛

进入 S4 前，必须确认：

- 同步失败不会影响本地使用。
- 冲突处理策略可解释。
- 设备权限不会自动扩大。
- 远程任务执行默认关闭。

### 暂不做

- 不做完全实时协同。
- 不要求同步全部原始文件。
- 不做跨端自动执行高风险操作。
- 不做复杂局域网穿透和公网暴露，除非有明确安全方案。

## S4 受控自主执行

### 阶段目标

让管家从“记录和展示”进入“主动发现和低风险执行”。核心是候选意图、自主任务、权限审批和审计闭环。

### 用户价值

系统能主动提醒用户可能遗漏的事，并在低风险范围内替用户完成整理、摘要、归档、生成草稿等动作。

### 自主能力范围

- 发现候选任务。
- 生成任务草稿。
- 自动整理低风险资料。
- 生成摘要和复盘。
- 创建提醒。
- 生成检查清单。
- 运行只读诊断。
- 执行用户预先批准的低风险工作流。

### 权限策略

- 默认只建议，不执行。
- 用户可按任务类型开启自动执行。
- 高风险操作必须确认。
- 对外发送、删除、购买、付款、提交、推送、系统设置变更默认禁止自动执行。
- 每次自主执行必须写入审计日志。

### 关键交付物

- [S3/S4 运行与验证基线](personal-ai-steward-s3-s4-runtime.md)。
- 自主任务规则中心。
- 候选意图评分。
- 任务建议解释。
- 权限审批组件。
- 模拟执行模式。
- 自主执行审计日志。
- 自主提案状态机和审批联动规则。
- 一键暂停自主执行。
- 错误候选任务批量清理入口。

### 验收标准

- 用户能看到每个自主任务的触发原因。
- 用户能把某类建议设置为自动通过、每次确认或永不建议。
- 低风险任务执行后有明确结果和影响范围。
- 高风险任务只生成计划，不直接执行。
- 已执行或已忽略的自主提案不能被再次批准或再次执行。
- 审批通过不能绕过高风险、A4 及以上或 `never` 策略阻断。
- 一键暂停后不再创建或执行自主任务。

### 阶段门槛

进入 S5 前，必须确认：

- 权限策略已经覆盖所有工具调用。
- 自主执行失败有恢复和报告。
- 用户可以批量清理错误候选任务。
- 审计日志足够支持排查“它为什么这么做”。

### 暂不做

- 不做无人值守高风险自动化。
- 不做自动发送邮件或聊天消息。
- 不做自动修改重要系统配置。
- 不做财务、医疗、法律类自动决策。

## S5 子 Agent 与工具生态

### 阶段目标

让主 Agent 不再单独处理所有任务，而是通过子 Agent、工具注册、能力声明和证据包机制完成复杂工作。

### 用户价值

用户可以把更复杂的问题交给系统处理，例如项目诊断、资料整理、任务拆解、生活计划、系统维护，但仍然能看到每个子 Agent 做了什么。

### 子 Agent 目标

- 开发 Agent：代码、构建、测试、错误诊断。
- 系统 Agent：服务、端口、磁盘、备份、进程。
- 文档 Agent：总结、归档、Runbook、知识条目。
- 生活 Agent：账单、证件、家务、旅行、健康记录。
- 项目经理 Agent：目标、任务、阻塞、里程碑。
- 审计 Agent：风险、权限、证据、不确定性。

### 工具体系目标

- 工具注册。
- 工具权限声明。
- 工具输入输出 schema。
- 工具执行日志。
- 工具健康检查。
- 工具版本和禁用。
- 工具结果进入任务、时间线、知识库或记忆库。

### 关键交付物

- 子 Agent 注册表。
- 工具能力注册表。
- Agent 调度器。
- 任务上下文包格式。
- 证据包格式。
- 审计 Agent。
- 多步骤任务执行页面。

### 验收标准

- 用户能看到一个任务调用了哪些子 Agent 和工具。
- 子 Agent 不能访问超出授权的上下文。
- 工具执行失败能返回结构化错误。
- 审计 Agent 能阻止或降级高风险任务。
- 多步骤任务可以暂停、取消、恢复。

### 阶段门槛

进入 S6 前，必须确认：

- 子 Agent 调度不会绕过权限中心。
- 工具能力声明可被 UI 展示和用户理解。
- 长任务可以持久化，不依赖前端页面保持打开。
- 证据包足够支持用户复盘。

### 暂不做

- 不做无限制的插件自动安装。
- 不允许子 Agent 自行扩大权限。
- 不让模型直接接触明文凭据。
- 不做没有审计日志的后台工具调用。

## S6 创造性智能层

### 阶段目标

在稳定底座上实现更有创造性的能力：上下文传送门、人生/项目仪表盘、自动生成工具、多代理会议、长期反省和个人数字孪生式草稿。

### 用户价值

系统不只是帮用户记录和执行任务，而是能持续理解用户的长期轨迹、减少重复劳动、辅助重大决策，并跨设备恢复完整工作状态。

### 重点能力

#### 个人时间线重建增强

- 自动聚合一天的工作段。
- 识别项目推进、停滞和上下文切换。
- 生成日总结、周总结和项目摘要。
- 从时间线恢复上下文。

#### 上下文传送门

- 保存工作现场快照。
- 跨设备恢复链接、文件、任务、项目和终端路径。
- 根据设备能力推荐执行位置。
- 防止敏感页面和命令自动打开。

#### 人生/项目仪表盘

- 今日必须处理。
- 项目推进。
- 生活维护。
- 记忆待确认。
- 系统健康。
- 风险队列。

#### 记忆反省机制

- 过期记忆检查。
- 冲突记忆对比。
- 低置信度记忆确认。
- 敏感记忆清理。
- 全局记忆和项目局部记忆迁移。

#### 自动生成工具

- 识别重复操作。
- 建议生成脚本、模板、页面或工作流。
- 草稿区审核。
- 首次执行确认。
- 工具版本回退。

#### 多代理会议

- 主持 Agent 定义问题。
- 专家 Agent 提方案。
- 反方 Agent 找风险。
- 审计 Agent 查证据。
- 执行 Agent 拆任务。

### 关键交付物

- 上下文快照模型。
- 跨设备继续工作入口。
- 人生/项目仪表盘。
- 记忆反省队列。
- 自动工具草稿区。
- 多代理会议页面。
- 决策记录沉淀机制。

### 验收标准

- 用户能从另一台设备继续某个项目现场。
- 仪表盘默认只展示需要行动的事项。
- 系统能主动发现一类重复工作并提出工具生成建议。
- 多代理会议能输出分歧、证据、风险和下一步任务。
- 记忆反省能发现并处理过期或冲突记忆。

### 阶段门槛

进入 S7 前，必须确认：

- 高级能力不会绕过已有权限、同步和审计体系。
- 自动生成工具有明确禁用和回退方式。
- 多代理会议结果不会自动执行高风险决策。
- 仪表盘没有变成噪音聚合页。

### 暂不做

- 不做完全替代用户决策的数字分身。
- 不做不可解释的长期画像。
- 不做自动对外表达用户意见。
- 不做无需确认的重大生活、财务、法律、医疗决策。

## S7 稳定化与长期运营

### 阶段目标

把系统从“能力丰富”提升到“长期可信”。重点是备份恢复、性能、升级迁移、数据治理、权限巡检和故障自愈。

### 用户价值

用户可以长期依赖它，而不用担心数据丢失、权限漂移、记忆污染、同步混乱或后台失控。

### 关键能力

- 自动备份和恢复演练。
- 数据库迁移和版本兼容。
- 设备撤销和密钥轮换。
- 采集噪音治理。
- 记忆质量评分。
- 长期任务归档。
- 审计日志压缩和保留策略。
- 性能监控。
- 故障自愈。
- 升级回滚。

### 关键交付物

- 备份恢复文档。
- 数据迁移文档。
- 权限巡检页面。
- 记忆健康页面。
- 同步健康页面。
- 性能和存储占用页面。
- 长期归档策略。

### 验收标准

- 能从备份恢复到可用状态。
- 升级失败可以回滚。
- 用户能查看权限是否漂移。
- 用户能清理长期无用数据。
- 系统能报告自身健康问题。

### 暂不做

- 不做企业级多用户权限系统。
- 不做团队协作平台。
- 不做公共 SaaS 化。
- 不做无边界插件市场。

## 阶段依赖关系

```text
S0 产品边界与安全基线
  -> S1 单机本地管家原型
    -> S2 个人数据底座
      -> S3 三端常驻与同步
        -> S4 受控自主执行
          -> S5 子 Agent 与工具生态
            -> S6 创造性智能层
              -> S7 稳定化与长期运营
```

其中 S7 不是最后才开始。备份、审计、权限巡检和数据治理要从 S1 起逐步引入，只是在 S7 阶段集中补齐长期运营能力。

## 横向能力轨道

每个阶段都要同时关注以下横向轨道。

| 轨道 | S1 目标 | S3 目标 | S6 目标 |
|---|---|---|---|
| 数据 | 本地事件和任务 | 跨端同步模型 | 长期记忆和知识治理 |
| 权限 | 最小权限配置 | 设备级权限 | Agent/工具级细粒度权限 |
| 审计 | 本机日志 | 跨端操作记录 | 高级自主行为解释 |
| UI | 工作台入口 | 设备与同步面板 | 时间线、仪表盘、会议、工具生成 |
| Agent | 基础对话 | 跨端上下文 | 子 Agent、多代理会议 |
| 自动化 | 手动任务 | 低风险自主任务 | 自动工具生成和工作流 |
| 安全 | 本地删除和暂停 | 加密同步和设备撤销 | 权限巡检和长期治理 |

## 优先级原则

如果资源有限，优先顺序如下：

1. 权限和审计优先于高权限工具。
2. 数据模型优先于复杂 UI。
3. 手动任务优先于自主任务。
4. 时间线和来源追踪优先于长期记忆。
5. 单机稳定优先于三端同步。
6. 低风险自动化优先于高风险自动化。
7. 子 Agent 调度优先于多代理会议。
8. 备份恢复优先于高级创造性能力。

## 第一条可执行路线

建议先按以下顺序落地，不直接跳到完整智能管家：

1. 创建当前项目中的管家工作台路由。
2. 建立本地 Agent 心跳和状态接口。
3. 建立事件、任务、审计日志三张核心表。
4. 做手动任务和事件列表。
5. 接入低风险采集：剪贴板摘要、指定目录文件事件、手动链接导入。
6. 做时间线基础视图。
7. 做候选意图池。
8. 做记忆库最小版本。
9. 做权限中心和一键暂停自主能力。
10. 再开始三端 Agent 和同步。

## 不可提前跨越的门槛

- 没有审计日志前，不接入高权限命令执行。
- 没有权限中心前，不做自主任务执行。
- 没有来源追踪前，不做长期记忆自动写入。
- 没有删除和导出前，不扩大数据采集。
- 没有冲突处理前，不做多端双向写入。
- 没有工具能力声明前，不做子 Agent 自动调度。
- 没有回退机制前，不做自动生成工具的正式启用。

## 阶段完成定义

一个阶段完成，不是指所有想象能力都实现，而是满足以下条件：

- 用户能在真实工作流中使用本阶段核心能力。
- 本阶段关键数据能保存、查询、导出和删除。
- 本阶段新增权限有配置和审计。
- 本阶段失败场景有明确提示。
- 本阶段能力不会破坏上一阶段已经稳定的能力。
- 下一阶段依赖的数据、权限和 UI 入口已经准备好。
