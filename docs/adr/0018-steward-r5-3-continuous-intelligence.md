# ADR-0018：R5.3 持续主动个人智能内核

- 状态：Accepted
- 日期：2026-07-19
- 决策范围：Windows 常驻采集、活动批次、长期画像、周期报告和自适应提醒

## 背景

R4.8 已经可以采集活动并由模型作出主动决定，R4.9 已经可以持久化执行多轮原生工具调用。但旧实现仍可能把单条 Observation 直接派发给模型，既产生大量孤立摘要，也不能稳定形成近期画像、长期画像、日报和提醒学习闭环。主服务作为 Windows 服务运行在 Session 0，不能可靠读取登录用户桌面；只检查后台循环是否存活，也不能证明真实活动仍在持续进入系统。

## 决策

### 1. 进程与权限边界

- 登录用户窗口、AFK、浏览器、剪贴板、截图和 UI 状态由 Session Companion 采集。
- 受限 LocalService 主服务负责持久化、聚合、模型调用和调度。
- 需要机器级权限的操作继续由 LocalSystem Privilege Broker 执行。
- “默认高权限开启”表示整套组件安装后自动运行且能力可达，不表示把所有进程都提升为 LocalSystem。

### 2. 批次替代逐事件模型派发

- 原始 Observation 幂等写入本地，并通过 heartbeat、AFK 和上下文切换聚合成 Activity Session。
- D0-D6 只作为历史记录兼容字段，不再作为 Companion 队列或主服务采集的允许列表；默认状态接口仍返回全部旧标签，供旧客户端正常展示。
- 明确凭据在进入 Companion 加密队列和主服务持久化前执行字段级本地脱敏：只替换命中的摘要片段、敏感键值、实体字段或文本证据，保留同一活动中的应用名、时间、上下文和其他真实证据。
- 单独出现的高熵字符串、哈希、UUID、构建 ID 或窗口标题不视为凭据，不能因此丢弃整条活动。无法解析的文本证据只移除该证据正文并保留 Observation 包络和脱敏来源。
- 默认每 30 分钟或活动边界形成不可变 Activity Batch。
- 模型只读取结构化批次摘要和证据引用，需要时再调用工具取证。
- 新批次使用 R4.9 Episode 循环；工具失败作为真实结果返回模型，服务重启后从持久回合继续。
- 新路径不再为每个普通 Observation 创建一次模型请求。历史 dispatch 和 insight 保持只读兼容。

### 3. 统一持久后台任务

Activity Batch、画像归纳、日报、周报和月报都使用可租赁任务：

- `status`、`due_at`、`next_attempt_at`
- `lease_owner`、`lease_expires_at`
- `attempt_count`、`checkpoint`
- `idempotency_key`、`control_generation`
- `episode_id`、`provider_response_id`
- `error_code`、`error_summary`

领取使用 `FOR UPDATE SKIP LOCKED`。完成、失败、取消和重试必须同时比较状态、租约或 control generation；旧 worker 不得覆盖暂停、取消或后续版本。

### 4. 分层记忆与报告

- 近期画像、稳定画像和用户明确事实分别版本化；合并视图使用 explicit > stable > recent 的优先级。
- 稳定画像默认需要来自至少三个不同日期的证据；用户纠正立即创建优先版本。
- 日报、周报、月报是持久实体。模型可以选择不通知，但不能通过沉默跳过报告写入。
- 报告和画像只引用批次、Session 和证据，不把无来源的模型文字当作事实。

### 5. 提醒学习闭环

- 通知反馈规范化为 opened、acted、acknowledged、snoozed、dismissed、ignored、cancelled 和 auto_resolved。
- 模型可以维护提醒类别的时段、频率、渠道和文案策略。
- 确定性投递控制器只保证去重、修订隔离、冷却、软预算、安静时段和投递可靠性，不使用固定业务规则决定用户需要什么。
- 提醒优化目标是帮助收益减去打扰、重复和错误成本；每轮始终允许不通知。

### 6. 模型与工具边界

- 普通对话和后台 Episode 均使用原生工具调用循环。
- 每轮注入真实采集新鲜度、批次积压、最近报告、画像版本、后台任务和通知状态。
- 自省工具必须查询当下数据库和循环状态，不得根据工具清单或旧记忆猜测。
- 不使用 D0-D6 或 A0-A9 决定模型可见内容或业务动作。Schema、幂等、签名、进程隔离、Watchdog、证据、暂停和急停仍是不可关闭的技术完整性约束。
- 旧数据策略或可选 PII 服务不可阻断活动采集；可选脱敏服务不可用时记录诊断并继续。明确凭据原文禁止落盘这一内容完整性规则仍然保留，且与旧等级无关。

## 默认调度

- 登录会话采样：10 秒。
- 会话化与批次控制器：1 分钟。
- 常规模型批次：30 分钟；活动边界宽限：5 分钟。
- 日报兜底：21:30；周报：周日 21:45；月报：每月首日 22:00。
- 后台 Episode 默认最多 64 轮、256 次工具调用、2 小时；`0` 表示不限制时仍保留无进展检测。
- 最近画像窗口：14 天；稳定画像默认最少 3 个证据日。

## 迁移与兼容

- 安装或升级时幂等创建 R5.3 表和索引，保留现有 Observation、Insight、Habit、Memory、Proactive Run 和无 Episode 的历史执行。
- 启用 batch 模式后，未完成的旧逐 Observation dispatch 标记为被新管线替代；已完成历史不删除。
- 启动时补建当前活动批次、最近 30 天画像归纳任务和最近 7 天报告缺口。
- Companion 离线时本地队列继续缓冲，成功落库后才确认删除。

## 可观测性和验收门禁

状态必须分别显示循环存活、来源新鲜度、队列健康、模型健康和最近结果。发布验收至少覆盖：

1. 不打开网页时真实窗口活动仍进入数据库并形成 Session。
2. 一小时多条 Observation 只形成预期数量的模型批次。
3. 模型中断、服务重启、租约过期和取消不会重复副作用或覆盖终态。
4. 每日报告即使选择静默也会持久化。
5. 通知反馈能改变后续 receptivity window 和提醒策略。
6. 用户询问后台状态时，模型通过实时自省工具返回与数据库一致的答案。

## 结果

该决策会增加数据库实体、后台控制器和状态界面复杂度，但把“能力列表”变成了可证明正在运行、能从长期反馈持续改进的个人智能系统。模型负责业务判断；确定性代码负责真实事实、持久恢复和副作用完整性。
