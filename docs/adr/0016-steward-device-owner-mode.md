# ADR-0016：设备所有者模式取代 D/A 业务门禁

状态：已接受
日期：2026-07-17

## 背景

早期 Steward 使用 `D0-D6` 数据等级和 `A0-A9` 操作等级同时承担分类、模型可见性、自动化策略、审批和执行门禁。该设计导致同一句自然语言在进入模型前被裁剪，也让模型已选择的真实工具在多个后置层被重复拒绝。用户要求管家以设备所有者身份工作：完整理解上下文，自主选择工具，并在本机或受信任设备上真实执行。

## 决策

生产 `steward run` 默认启用 `STEWARD_OWNER_MODE=true`。

在设备所有者模式中：

- 模型可读取当前对话、长期记忆、活动、习惯、事件、任务和设备上下文，不按 D 等级裁剪或脱敏。
- 模型得到完整 Runtime ToolSpec，并通过原生 function calling 自主决定调用顺序和参数。
- A 等级、权限 ceiling、数据 ceiling、旧自动化策略和模型最高数据等级不再阻止工具选择、单 Run、多 Agent、主动执行或跨设备普通执行。
- 本机文件根覆盖现有卷，Shell 可解析任意已安装可执行文件，Web 工具可访问本机和私有网络。
- 主动内核可以基于全部活动和长期上下文决定静默、询问、提醒、创建任务或执行帮助。
- D/A 字段只允许留在无法原地迁移的历史存储、审计和已签名协议载荷中；数据策略、权限策略、设备权限、审批规则和受限配置工具在 owner mode 启动时全部删除。公开的模型设置、Runtime 工具、自治动作和设备契约不再返回这些上限字段。新代码不得把历史字段当作授权源。

以下机制不是 D/A 业务门禁，继续保留：

- 工具名与参数 schema 校验、超时、资源配额、幂等和结果证据。
- 全局暂停/急停、watchdog、租约和 Job Object 进程生命周期控制。
- 跨设备身份、信任、消息签名、时效与结果验签。
- 独立 Privilege Broker 的固定 capability、Windows 服务身份、凭据不出 Broker 和防重放协议。是否需要 Broker 由 `privilege.execute` 工具本身决定，不再由 A 数字推断。

## 后果

管家不会再因为消息或上下文的 D 标签回答“无权查看”，也不会因为计划的 A ceiling 拒绝一个已注册工具。旧对话和由旧对话生成的执行、任务与记忆可以事务性清空，且限制策略不会在重启后复活。模型可以更充分地利用设备，但错误工具调用的影响面也更大；因此工具参数校验、进程隔离、证据、签名和全局急停必须保持可用，且不能被 owner mode 关闭。

如果未来需要多用户或企业隔离，应建立基于主体、设备和 capability 的真实授权模型，而不是恢复 D/A 数字等级作为模型前置过滤器。
