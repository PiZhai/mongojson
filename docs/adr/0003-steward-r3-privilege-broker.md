# ADR-0003：R3 独立 Privilege Broker

- 状态：Accepted
- 日期：2026-07-16
- 决策范围：A4-A7 高权限本机能力、统一停止、执行证据与服务部署

## 背景

R2/R2.6 的 Runtime 与 S4 已能执行真实操作，但进程仍运行在主 Steward 身份中。继续在主进程里增加管理员/root 权限会把自然语言解析、网络 API、数据库、模型调用和系统提权放进同一个信任域；任一输入验证或依赖缺陷都可能直接变成整机权限。

R3.0 要解决的不是“允许任意命令提权”，而是把少量、事先登记、可审计的高权限动作移入独立且更小的执行边界。

## 决策

新增独立可执行程序 `steward-broker`，以 system scope 服务运行。主 Steward 保持普通用户身份，通过仅回环监听的版本化 HTTP 协议请求能力。Broker 是可执行路径、固定参数、工作目录、超时、输出上限和二进制哈希的唯一执行授权源；主进程在执行请求中只能提交稳定能力名，不能提交路径、参数、环境变量或命令文本。

### 信任边界

```text
用户 / Runtime / S4
        │ 计划哈希、审批引用、统一停止代际
        ▼
普通用户 Steward 主进程
        │ HMAC 请求认证 + 时间戳 + 单次 nonce
        │ 能力名，不含命令路径或动态参数
        ▼
system scope Privilege Broker
        │ 固定策略 + executable SHA-256 再校验
        │ 单次短时能力令牌 + 进程树托管
        ▼
受限 A4-A7 操作系统动作
        │
        └── Ed25519 签名 receipt + 签名哈希链审计
```

### 协议规则

- R3.0 基线 API 版本为 `steward-privilege-broker/v1`，只接受 `localhost` 或 loopback IP，客户端禁用代理继承；后续 R3.2/R3.3 分别升级到 v1.1/v1.2，客户端必须精确匹配版本。
- 每个请求以共享 HMAC-SHA256 密钥认证，签名覆盖 method、path、Unix 时间戳、随机 nonce 和 body SHA-256。
- Broker 在时钟偏差窗口内拒绝重复 nonce，阻断请求重放。
- Broker 状态、能力令牌和执行 receipt 均由固定 Ed25519 身份签名；主进程只保存并校验公钥。
- grant 绑定能力策略摘要、调用主体、计划哈希、审批引用、控制代际、Broker 实例和过期时间。
- 能力令牌只有一次使用机会。Broker 重启会更换实例 ID，使重启前未使用令牌自动失效。
- 每次执行前重新计算可执行文件 SHA-256；文件发生变化即拒绝启动。
- 子进程获得经过清洗的最小环境，不能继承 Broker 密钥。Windows 使用 Job Object，Unix 使用 process group 回收整棵进程树。

### 策略边界

R3.0 只接受 A4-A7、`high` 或 `critical` 的固定能力。以下输入在 Broker 策略加载时直接拒绝：

- A0-A3 用户态能力；
- A8/A9 凭据、财务或不可代理权限；
- 相对可执行路径、缺失或不匹配的 SHA-256；
- `cmd.exe`、PowerShell、`sh`、Bash、脚本宿主等通用解释器；
- 动态参数、动态工作目录、动态环境变量；
- 超过策略边界的超时、输出大小或能力数量。

数据库中的 S4 tool definition 只保留管理元数据。执行时其权限、风险、可执行文件名和参数数量必须与 Broker 公开的签名能力元数据一致；实际路径和参数不会从数据库传给 Broker。回滚必须登记为独立的 `<capability>.rollback` 能力。

### 统一紧急停止

PostgreSQL 中的统一控制状态仍是产品控制面的权威记录。每次真实 stop/resume 递增 generation；主进程把相同状态同步到 Broker。Broker 持久化自身停止状态并执行以下 fencing：

- stopped 时拒绝发 grant 和执行；执行准入与 stop 使用同一读写锁线性化，stop 必须取消已登记动作或先阻止其登记；
- stop 立即取消所有 Broker 托管进程树；
- 旧 generation 的 grant 在恢复后仍不可使用；
- 重复的同状态、同代际同步是幂等操作，不产生重复审计。

若数据库与 Broker 状态不一致，控制 API 和工作台显示错误，不能把 Broker 离线误报为安全运行。主 Steward 仍可提供记录、查询和审批查看等非执行功能。

### 证据与审计

Broker 返回 Ed25519 签名 receipt，包含 execution ID、能力及策略摘要、主体、计划哈希、审批引用、控制代际、开始/结束时间、退出码、成功状态、错误码、`audit_persisted`、stdout/stderr 大小、SHA-256 与截断标记。正文受能力输出上限和 R2.6 Evidence 治理二次约束。`execute.started` 审计未持久化时绝不启动进程；动作已经发生但 finished 审计失败时仍返回签名 receipt，并把 Run 判为失败且保留治理后的故障证据，避免把未知结果误报为成功。

Broker 独立写入 JSONL 审计链；每条记录包含单调 sequence、上一条 hash、本条 SHA-256 和 Ed25519 签名。启动时会完整验证已有链，发现篡改则拒绝启动。

## 明确限制

- R3.0 的 Broker 独立验证请求来源、策略、令牌、代际和二进制，但审批事实仍由主 Steward 声明。该缺口已由 [ADR-0004：R3.1 独立审批证明](0004-steward-r3-1-independent-approval.md) 关闭：Broker 现在要求独立 Authority 签名的短时一次性票据。
- HMAC 客户端密钥允许主进程请求策略内任一能力，因此 Broker 策略必须保持最小能力集，不能登记通用 Shell 或可注入参数的包装器。
- R3.0 不托管明文凭据，不实现 A8/A9 密钥代理，也不负责跨设备委派。
- system scope 只提供进程身份隔离。生产部署必须让管理员/root 控制 Broker 二进制、策略、私钥、状态和审计文件的 ACL；普通 Steward 用户不得拥有写权限。

## 后果

正面结果：主进程不再直接获得高权限；高权限攻击面缩小为固定协议与固定能力；令牌、进程、停止和审计均可被独立验证。

成本：部署需要第二个服务、独立密钥与受保护策略；升级能力必须重新计算二进制哈希并由管理员更新策略；Broker 不可达时高权限能力安全失败。

## 被否决方案

- 主进程直接以管理员/root 运行：信任域过大，任何主进程缺陷都拥有整机权限。
- 每次执行 `sudo`、UAC 或 shell：命令和参数边界不可稳定验证，难以阻断注入与重放。
- 只用数据库 tool definition 做白名单：数据库属于普通主进程信任域，不能充当高权限执行授权源。
- 长期 bearer token：泄露后可重放，无法绑定计划、审批、代际和单次执行。
